# [백엔드지식] - 2026-06-01

## 📌 문제

서버 측 캐싱 전략인 **Cache-Aside**, **Read-Through**, **Write-Through**, **Write-Behind(Write-Back)** 4가지의 동작 방식을 설명하고, 각 전략의 장단점과 적합한 사용 시나리오를 비교하시오. 또한 **캐시 스탬피드(Cache Stampede)**와 **캐시 오염(Cache Pollution)** 문제가 무엇인지, 이를 방지하기 위한 방법도 함께 서술하시오.

## 💡 힌트

각 전략이 캐시 미스(Cache Miss) 발생 시 데이터를 어떻게 처리하는지, 그리고 데이터 **쓰기 시점과 대상**(캐시만? DB만? 동시에?)을 기준으로 전략들을 구분해보세요. 캐시와 DB 사이의 **데이터 일관성**과 **성능** 사이의 트레이드오프를 중심으로 생각하고, 동시에 수많은 요청이 만료된 캐시 키에 몰릴 때 어떤 일이 벌어지는지도 상상해보세요.

## ✍️ 내 답변 (2026-06-01 제출)

- **Cache-Aside**: 캐시를 먼저 조회하고 캐시에 데이터가 없으면 데이터베이스를 조회하는 전략. 저장 시점은 서버가 DB에서 직접 조회 후, cache에 저장. 캐시 채우는 코드를 앱이 매번 직접 써야 함. 보통 읽기 위주 범용.
- **Read-Through**: 캐시를 조회 시, 조회 내용이 없다면 캐시가 DB를 조회 후 저장하고 반환함. 캐시 계층이 죽으면 읽기 경로 자체가 막힐 수 있음. 읽기 위주 + 로딩 로직 중앙화.
- **Write-Through**: 쓰기 시, 데이터베이스와 캐시 둘 다 쓰기를 실행함. 쓰기가 2배로 늘어나서 쓰기가 느려짐. 정합성 중요, 쓰고 바로 읽는 패턴(프로필·설정·계정).
- **Write-Behind**: 쓰기 시 캐시로 저장되었음을 알린 후, 이후에 비동기로 DB에 저장. 메시지 유실 위험이 있음. 쓰기 폭주·유실 허용 가능한 경우 (카운터·로그·조회수·집계).

**캐시 스탬피드(Cache Stampede)**: 인기 있는 캐시 키의 TTL이 만료되는 순간, 동시에 들어온 수많은 요청이 한꺼번에 미스를 일으켜 전부 DB로 몰리는 현상. DB가 과부하로 마비될 가능성 존재.

**캐시 오염(Cache Pollution)**: 거의 다시 조회되지 않을 데이터가 캐시를 차지하면서, 정작 자주 쓰이는 데이터를 밀어내 적중률을 떨어뜨리는 현상.

**해결 방법**
- 캐시 스탬피드: Mutex Lock (최초 1명에게만 DB 조회 권한, 나머지는 캐시 대기) / Probabilistic Early Expiration (만료 전 미리 갱신)
- 캐시 오염: TTL (시간제 만료) / LRU (오래 안 쓴 항목 제거) / LFU (적게 쓰인 항목 제거) 정책으로 방지
