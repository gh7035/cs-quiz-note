# 2026-06-08 - 백엔드지식: JWT vs 세션 인증 방식 비교

## 📌 문제

JWT(JSON Web Token)와 세션(Session) 기반 인증 방식의 동작 원리를 각각 설명하고, 두 방식의 장단점을 비교하시오. 또한 JWT 토큰 탈취(Token Hijacking) 문제와 세션 기반 인증의 서버 수평 확장(Scale-out) 문제를 각각 어떻게 해결하는지 서술하시오.

## 💡 힌트

두 방식의 핵심 차이는 인증 상태를 **어디에 저장하는가**에 있습니다. 상태가 서버 측에 있는지, 아니면 클라이언트가 소지하는 토큰 자체에 담겨 있는지를 기준으로 Stateful과 Stateless의 개념을 떠올려보세요. 서명(Signature)으로 위변조는 막을 수 있지만, 유효한 토큰이 탈취됐을 때 서버가 이를 즉시 무효화하기 어려운 이유와 Access Token / Refresh Token 분리 전략이 이 문제를 어떻게 완화하는지 생각해보세요. 세션 방식에서 여러 서버 간 세션 불일치 문제를 해결하기 위한 방법(세션 클러스터링, 외부 세션 저장소)도 함께 고민해보세요.

## 💬 내 답변

jwt 기반 인증은 header에 jwt의 서명이 손상되지 않았으면 내용도 손상되지 않았다고 판단하고, 인가를 허가하는 원리이고, 세션 기반 인증은 stateful의 원칙을 유지하기 위해 나온 개념으로, jwt와 다르게 한번의 요청마다 session을 DB에서 확인한뒤, 인가를 허가하는 원리이다. jwt는 stateless의 방식이고, 별도의 db조회를 거치지 않아도 되어서 서버 부하가 감소하지만 세션보다 탈취가 쉽다는 문제점이 존재하고, session은 stateful의 원칙을 지키기 때문에 보안성이 jwt보다 뛰어나다는 장점이 있지만, 한번의 인증마다 DB를 반복적으로 조회하기 때문에 서버부하가 커질 수 있다.

각각의 단점을 보완하기 위해서 나온 개념은 jwt에서는 refresh token을 별도로 발급하고, access token의 만료기간을 짧게 설정함으로써 탈취가 되더라도 refresh token으로 재발급을 받아야 하는 경우를 생기게 함으로써 단점을 극복할 수 있다. 덧붙여서, refresh token을 재발급 받을때 재발급한 refresh token 또한 재발급 하는 전략도 존재하고, 또한 httponly cookie에 token을 담아서 xss 공격으로부터 예방이 가능하다. 또한 blacklist에 토큰을 추가해서 토큰을 무효화 시킬수 있는 방법 또한 존재한다.

session에서는 redis 같은 in-memory DB를 서버들의 공용으로 사용함으로써 세션 불일치를 막는다. 또한 세션 클러스터링 이란 기술도 존재하는데 서버의 데이터를 실시간으로 복제하는 기술이다. 복제할수록 오버헤드가 커진다는 단점이 있다.

## 📖 정답 및 해설

### JWT(JSON Web Token) 인증

**동작 원리:**
- 로그인 시 서버가 `Header.Payload.Signature` 형태의 토큰 발급
- 클라이언트가 토큰을 저장하고 매 요청마다 `Authorization: Bearer <token>` 헤더에 포함
- 서버는 서명(Signature)만 검증 → DB 조회 불필요 (Stateless)

**장점:**
- DB 조회 없이 서명 검증만으로 인증 → 서버 부하 감소, 빠른 응답
- 수평 확장(Scale-out)에 유리 — 어느 서버에서도 동일하게 검증 가능

**단점:**
- 토큰 탈취 시 만료 전까지 서버가 즉시 무효화 불가
- Payload는 Base64 인코딩이므로 누구나 디코딩 가능 → 민감 정보 저장 금지

### 세션(Session) 인증

**동작 원리:**
- 로그인 시 서버가 Session ID 생성 → 서버 측 저장소(DB, 메모리)에 보관
- 클라이언트는 쿠키로 Session ID만 유지
- 매 요청마다 서버가 저장소에서 세션 조회 (Stateful)

**장점:**
- 세션 삭제로 즉시 강제 로그아웃 가능 → 높은 보안성
- 서버가 인증 상태를 완전히 통제

**단점:**
- 매 요청마다 저장소 조회 → 서버 부하 증가
- 다중 서버 환경에서 세션 불일치 문제 발생

### JWT 토큰 탈취 해결 전략

| 전략 | 설명 |
|------|------|
| **Access/Refresh Token 분리** | Access Token 유효기간 짧게(5~15분) 설정, Refresh Token으로 재발급 |
| **Refresh Token Rotation** | 재발급 시마다 Refresh Token도 새로 갱신 → 탈취된 Refresh Token 자동 무효화 |
| **HTTPOnly Cookie** | JavaScript로 접근 불가한 쿠키에 토큰 저장 → XSS 공격 방어 |
| **토큰 블랙리스트** | Redis 등에 무효화된 토큰 저장 → 강제 로그아웃 구현 |

### 세션 Scale-out 해결 전략

| 전략 | 설명 |
|------|------|
| **외부 세션 저장소(Redis)** | 모든 서버가 공유하는 중앙 세션 저장소 → 세션 불일치 해소 |
| **세션 클러스터링** | 서버 간 세션 데이터 실시간 복제 (단, 복제 오버헤드 발생) |
| **Sticky Session** | 동일 사용자 요청을 항상 같은 서버로 라우팅 (단, 서버 장애 시 세션 손실) |
