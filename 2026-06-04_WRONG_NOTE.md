# 2026-06-04 오답 노트 - 운영체제: 교착상태(Deadlock)

## 📌 문제

교착상태(Deadlock)가 발생하기 위한 4가지 필요 조건(Coffman Conditions)을 설명하고, 교착상태를 처리하는 세 가지 방법인 **예방(Prevention)**, **회피(Avoidance)**, **탐지/회복(Detection & Recovery)**의 핵심 개념과 각각의 트레이드오프를 서술하시오.

---

## ✍️ 내 답변

4가지 조건 : 상호배제, 점유와 대기, 비선점, 순환 대기

예방 시 : 사전에 교착상태를 방지함으로 안정성이 뛰어나지만, 실시간 감지를 함으로 전체적으로 시스템 이용률이 저하된다.

회피 시 : 자원을 할당할 때 마다 탐지함으로, 오버헤드가 발생한다.

탐지/회복 시 : 교착상태를 탐지하기 위한 주기적인 알고리즘 실행으로 회피와 마찬가지로 추가적인 오버헤드가 발생. 회복 과정에서 일부 프로세스가 강제 종료되거나 작업 내역이 손실될 수 있어 작업의 연속성이 훼손될 수 있다.

---

## ✅ 정답

### 1. 4가지 필요 조건 (Coffman Conditions)

| 조건 | 설명 |
|------|------|
| **상호 배제 (Mutual Exclusion)** | 자원은 한 번에 하나의 프로세스만 사용 가능. 다른 프로세스는 해제될 때까지 대기 |
| **점유와 대기 (Hold and Wait)** | 최소 한 개의 자원을 점유한 채로 다른 자원을 추가 획득하기 위해 대기 |
| **비선점 (No Preemption)** | 점유된 자원은 해당 프로세스가 자발적으로 반납하기 전까지 강제로 빼앗을 수 없음 |
| **순환 대기 (Circular Wait)** | P1→P2→…→Pn→P1 형태로 프로세스들이 원형으로 서로의 자원을 기다림 |

---

### 2. 예방 (Prevention)

**핵심 개념**: 4가지 조건 중 하나를 **시스템 설계 수준에서 영구 제거**하여 교착상태 자체가 발생하지 못하게 함. "실시간 감지"와는 무관.

| 제거 조건 | 방법 |
|-----------|------|
| 상호배제 제거 | 자원을 공유 가능하게 변경 (일부 자원은 불가) |
| 점유대기 제거 | 실행 전 필요한 모든 자원을 한 번에 요청 |
| 비선점 제거 | 추가 자원 획득 불가 시 기존 자원 자발 반납 후 재요청 |
| 순환대기 제거 | 모든 자원에 전역 번호를 부여하고, 오름차순으로만 요청 허용 |

**트레이드오프**: 자원 이용률 저하, 기아(Starvation) 발생 가능

---

### 3. 회피 (Avoidance)

**핵심 개념**: 자원을 할당하기 **전에** 시스템이 "안전 상태(Safe State)"를 유지하는지 확인. 교착상태로 이어질 수 있는 할당 자체를 거부. "탐지(Detection)"와는 다른 개념.

- **은행원 알고리즘 (Banker's Algorithm)**: 프로세스의 최대 자원 요구량을 사전 파악 후, 할당해도 **안전 순서열(Safe Sequence)**이 존재하면 허용
- 안전 상태: 모든 프로세스가 최종적으로 자원을 획득하고 종료될 수 있는 순서가 존재하는 상태

**트레이드오프**: 최대 자원 요구량을 사전 선언해야 하며, 매 할당 시마다 안전 상태 검사 오버헤드 발생

---

### 4. 탐지/회복 (Detection & Recovery)

**핵심 개념**: 교착상태 발생을 허용하되, 주기적으로 탐지 알고리즘을 실행하여 발견 시 회복 조치

- **탐지**: 자원 할당 그래프(Resource Allocation Graph)에서 사이클 검출, 자원 요청 행렬 분석
- **회복**: 프로세스 강제 종료(kill), 자원 선점(preemption), 체크포인트/롤백

**트레이드오프**: 탐지 알고리즘 주기적 실행 오버헤드, 회복 과정에서 작업 손실 및 연속성 훼손 가능

---

## 🔍 틀린 이유 분석

### ❌ 예방(Prevention) 오류
- **내 답변**: "실시간 감지를 함으로 시스템 이용률이 저하"
- **오류 원인**: 예방(Prevention)은 **감지(Detection)와 완전히 무관**. 예방은 4가지 조건 중 하나를 시스템 설계 단계에서 원천 차단하는 방식이며, 런타임 감지가 아님.
- **정확한 표현**: "4가지 조건 중 하나를 원천 제거함으로써 교착상태 발생 자체를 불가능하게 하나, 자원 이용률 저하와 기아 문제가 트레이드오프"

### ❌ 회피(Avoidance) 오류
- **내 답변**: "자원을 할당할 때마다 탐지함으로 오버헤드 발생"
- **오류 원인**: "탐지(Detection)" 표현을 잘못 사용. 회피는 탐지가 아니라 **'안전 상태 확인'**. 은행원 알고리즘을 통해 할당 전 시스템이 안전 상태인지 검사.
- **정확한 표현**: "자원 할당 전 안전 상태(Safe State) 여부를 확인(은행원 알고리즘)하므로, 매 할당마다 검사 오버헤드와 최대 자원량 사전 선언이 필요"
