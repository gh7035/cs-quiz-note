## 2026-05-25 | 백엔드지식

### 📌 문제
JWT(JSON Web Token) 기반 인증과 세션(Session) 기반 인증의 동작 원리를 비교 설명하고, 각각의 장단점과 적합한 사용 사례를 서술하시오.

### 💬 내 답변
jwt 기반 인증은 stateless(무상태)를 유지하기에 용이하고, 서버에 따로 인증정보를 저장하지 않는다는 장점이 존재하지만, 탈취나 보안에 session보다 비교적 취약한 부분이 존재하고, session 기반 인증은 인증 하나하나마다 데이터베이스에서 인증 정보를 확인함으로, 보안성은 jwt 보다 비교적 좋지만, 트레픽이 많아질 시, 서버가 감당하는 부담이 늘어난다는 단점이 존재한다.

### ✅ 정답 및 해설

#### 동작 원리

**JWT 기반 인증 (Stateless):**
1. 사용자가 로그인하면 서버는 사용자 정보(Payload)를 비밀키로 서명한 JWT를 생성
2. 서버는 토큰을 클라이언트에게 반환 (서버 측에 별도 저장 없음)
3. 클라이언트는 토큰을 로컬 스토리지 또는 쿠키에 보관
4. 이후 요청 시 `Authorization: Bearer <token>` 헤더로 전송
5. 서버는 서명만 검증하면 인증 완료 → 저장소 조회 불필요

**세션 기반 인증 (Stateful):**
1. 사용자가 로그인하면 서버는 세션을 생성하고 세션 저장소(DB/메모리)에 저장
2. 서버는 세션 ID를 `Set-Cookie` 헤더로 클라이언트에 반환
3. 클라이언트는 이후 요청 시 쿠키에 세션 ID를 자동 포함하여 전송
4. 서버는 세션 저장소를 조회하여 유효성 검증

#### 장단점

| 구분 | JWT | 세션 |
|------|-----|------|
| **장점** | 서버 저장소 불필요, 수평 확장(Scale-out) 용이, 다양한 도메인 활용 가능 | 즉시 무효화 가능, 서버 측 제어 용이, 보안성 높음 |
| **단점** | 토큰 탈취 시 만료 전까지 무효화 불가, 토큰 크기 증가 가능 | 서버 부담(저장소 조회), 수평 확장 시 세션 공유 문제 발생 |

#### 적합한 사용 사례

- **JWT 적합**: 마이크로서비스 아키텍처, 모바일 앱, 분산 시스템, 외부 공개 API 서버
- **세션 적합**: 전통적인 웹 애플리케이션, 보안이 중요한 금융·의료 서비스, 즉각적인 로그아웃이 필요한 서비스

#### 오답 이유
장단점은 부분적으로 맞았으나, **동작 원리**(JWT 토큰 생성·서명·검증 흐름 / 세션 생성·세션ID 쿠키 전달 흐름)와 **적합한 사용 사례**가 전혀 서술되지 않았습니다.
